# Vibe Coding Development Rules

## 基本原则

- PRD 先行：所有功能变更必须能追溯到 `docs/PRD.md` 的目标、范围、验收标准或明确新增需求。
- 冲突先指出：如果实现中发现 PRD、代码、Chrome 权限、noon 页面结构或用户目标之间存在冲突，必须先记录并告知，不能静默绕开。
- 小步开发：每次变更尽量保持可验证、可回滚、可解释。
- 本地优先：v1 不引入后端，不上传用户页面数据。
- 完整评测：每次开发后必须运行 `npm run evaluate`，生成 `reports/evaluation-report.md`。
- 版本同步：每次功能更新或行为变更都必须同步升级 `manifest.json`、`package.json`、`evals/evaluation_set.json` 的版本号，并在 `CHANGELOG.md` 顶部倒序记录对应版本更新日志。

## 开发完成定义

一次开发只有同时满足以下条件才算完成：

- 实现符合 `docs/PRD.md`，没有未说明的范围偏移。
- `evals/evaluation_set.json` 中所有用例 100% 通过。
- 基础自动化测试 100% 通过。
- 已生成最新测试报告。
- 已同步版本号，并在 `CHANGELOG.md` 顶部写入本次版本的倒序更新日志。
- 如果发现 PRD 冲突，已经在 PRD、实现说明或报告中明确记录。

## 长期评测集维护

- 评测集文件：`evals/evaluation_set.json`。
- 每个新增功能必须新增或更新对应评测用例。
- 不允许为了通过评测而删除关键用例；确实需要调整时，要说明 PRD 依据。
- 评测覆盖至少包括：PRD 合规、销量解析、排序、导出、建议、权限和非 noon 页面处理。

## 标准开发流程

1. 读 PRD，确认变更目标属于范围内。
2. 如发现冲突，先指出并更新方案。
3. 实现最小可用变更。
4. 升级版本号并更新 `CHANGELOG.md`。
5. 更新或新增评测用例。
6. 运行 `npm run evaluate`。
7. 确认报告显示 100% 通过。
8. 汇报变更内容、版本号、评测结果和报告路径。
