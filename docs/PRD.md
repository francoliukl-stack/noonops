# Noon Sales Ops Copilot Chrome 插件 PRD v0.1

## Summary

面向 noon 电商平台销售运营人员，做一个运行在商品列表、搜索结果、店铺商品页上的 Chrome 插件。插件通过页面右侧浮窗读取当前页面可见商品信息，按商品销售量排序，并基于排序结果给出运营辅助建议，帮助用户快速判断哪些商品值得跟进、哪些商品需要优化、哪些商品可作为选品或竞品参考。

v1 聚焦“当前页面可见商品”的轻量分析，不做账号登录、不连接后端、不跨页自动抓取、不上传数据。

## Product Goals

- 在 noon 当前页面直接完成商品销量排序，不要求用户复制粘贴数据。
- 在插件浮窗中展示排序结果，减少离开页面或打开表格的操作。
- 自动识别高销量、低销量、潜力商品，并给出可执行的运营建议。
- 用低成本、快反馈的 v1 为后续价格监控、竞品追踪、批量导出、AI 运营助手打基础。

## Target Users

- Noon 平台卖家运营人员
- 选品和类目运营人员
- 竞品分析人员
- 需要快速浏览商品表现的小团队负责人

## Core User Stories

- 作为运营，我在 noon 商品列表页打开插件后，可以看到当前页面商品按销量从高到低排序。
- 作为运营，我可以在浮窗里看到商品名称、价格、销量、评分、排名、商品链接等关键字段。
- 作为运营，我可以快速识别当前页面销量最高的商品，并获得“可参考其标题、价格、促销方式”的建议。
- 作为运营，我可以看到低销量但价格或评分有潜力的商品，并获得优化建议。
- 作为运营，我可以一键复制或导出当前分析结果，方便发给团队或放入表格。

## V1 Scope

### Chrome 插件入口

- 使用 Chrome Extension Manifest V3。
- 插件图标点击后，在当前 noon 页面内打开一个右侧浮窗。
- 浮窗可关闭、刷新分析、切换排序方式。
- 仅在 noon 相关域名页面展示可用分析；其他网站显示“不支持当前页面”。

### 当前页面商品识别

插件从当前页面 DOM 中提取当前可见商品卡片信息：

- 商品名称
- 商品链接
- 商品图片
- 当前价格
- 原价或折扣信息，如果页面可见
- 销售量，如果页面可见
- 评分和评论数，如果页面可见
- 当前页面展示位置

默认只分析当前已经加载出来的商品，不自动翻页，不模拟滚动加载。

### 销量排序

- 默认按销售量从高到低排序。
- 支持切换：
  - 销量高到低
  - 价格低到高
  - 价格高到低
  - 评分高到低
  - 页面原始顺序
- 如果商品缺少销量字段，标记为“未识别销量”，排在有销量商品之后。
- 销量文本需要支持常见格式解析，例如 `100 sold`、`100+ sold`、`1.2K sold`、`10k+`。

### 浮窗分析结果

浮窗包含三块：

- 顶部概览：商品数量、识别到销量的商品数、最高销量、平均价格。
- 排序列表：每个商品展示排名、名称、价格、销量、评分、链接。
- 运营建议：基于当前页面商品表现生成简短建议。

### 辅助建议规则

v1 使用本地规则，不依赖大模型 API。

- 高销量参考：销量 Top 3 商品，建议观察标题关键词、价格区间、促销标签、主图风格。
- 价格带建议：识别高销量商品集中价格区间，提示当前页主流成交价格带。
- 低销量风险：价格高但销量低的商品，提示检查价格竞争力、主图、标题或促销。
- 评分机会：评分高但销量低的商品，提示可能需要增加曝光或优化关键词。
- 数据不足：如果页面销量字段不可见，提示当前页面无法完成销量排序，可改用评论数或评分作为辅助指标。

### 导出与复制

- 支持复制当前排序结果为表格文本。
- 支持导出 CSV。
- 字段包含：排名、商品名称、销量、价格、评分、评论数、商品链接。

## Non Goals For V1

- 不做自动登录或绕过 noon 权限限制。
- 不做跨页批量抓取。
- 不做后台数据库。
- 不做价格历史追踪。
- 不做自动下单、自动改价、自动发布商品。
- 不上传用户浏览数据到服务器。
- 不保证 noon 所有国家站点页面结构一次性全部兼容，先以当前主用页面为准。

## UX Requirements

- 浮窗宽度约 420px，固定在页面右侧。
- 不遮挡页面主要浏览区域过多，可折叠。
- 使用运营工具风格：信息密度高、清晰、少装饰。
- 商品列表要易扫读，排名、销量、价格必须突出。
- 对无法识别的字段明确显示为空或“未识别”，避免假数据。
- 操作按钮包括：刷新、复制、导出 CSV、关闭。

## Technical Approach

- Chrome Extension MV3。
- `manifest.json` 定义插件入口和最小权限。
- content script 读取页面商品卡片并注入浮窗。
- background service worker 在点击插件图标时注入脚本和样式。
- shared parser module 解析商品字段和销量文本。
- 数据处理全部在浏览器本地完成。
- DOM 解析采用可配置 selector 策略，避免把 noon 页面结构硬编码散落在 UI 逻辑中。
- 建议生成采用纯前端规则引擎，后续可替换为 AI 建议接口。

## Public Interfaces / Data Model

核心商品数据结构：

```ts
type ProductItem = {
  id: string;
  title: string;
  url: string;
  imageUrl?: string;
  price?: number;
  currency?: string;
  originalPrice?: number;
  discountText?: string;
  salesText?: string;
  salesCount?: number;
  rating?: number;
  reviewCount?: number;
  pagePosition: number;
};
```

核心分析结果：

```ts
type AnalysisResult = {
  pageUrl: string;
  analyzedAt: string;
  totalProducts: number;
  productsWithSales: number;
  sortedProducts: ProductItem[];
  insights: Insight[];
};
```

建议结构：

```ts
type Insight = {
  type: "top_seller" | "price_band" | "low_sales_risk" | "rating_opportunity" | "data_gap";
  title: string;
  message: string;
  relatedProductIds?: string[];
};
```

## Acceptance Criteria

- 在 noon 商品列表页点击插件后，可以打开页面内浮窗。
- 插件能识别当前页面可见商品，并展示至少商品名、价格、链接。
- 如果页面展示销量，插件能解析销量并按销量从高到低排序。
- 无销量商品不会导致插件报错，且会被清楚标记。
- 用户可以刷新分析结果。
- 用户可以复制排序结果。
- 用户可以导出 CSV。
- 插件不会修改 noon 页面原始数据或触发购买、登录等敏感操作。
- 在非 noon 页面打开时，插件显示不支持提示。

## Evaluation Policy

每次开发完成前必须运行长期评测集，并产出测试报告。只有 `evals/evaluation_set.json` 中全部用例 100% 通过，且基础自动化测试通过，才算开发完成。

如果开发过程中发现 PRD 与实际实现、noon 页面结构、Chrome 权限模型或用户目标存在冲突，必须先指出冲突并更新 PRD 或实现方案，不能静默偏离 PRD。

## Test Plan

- 使用 noon 搜索结果页测试：确认商品数量、排序、链接、价格识别正常。
- 使用 noon 店铺或类目页测试：确认不同列表结构下浮窗不崩溃。
- 使用无销量展示页面测试：确认出现“数据不足”建议。
- 使用销量文本样例测试解析：
  - `100 sold`
  - `100+ sold`
  - `1.2K sold`
  - `10k+`
  - 空值
- 测试复制结果是否能直接粘贴到表格。
- 测试 CSV 是否包含正确列名和 UTF-8 内容。
- 测试非 noon 页面不会注入错误 UI。
- 测试刷新后不会重复插入多个浮窗。

## Assumptions

- v1 只分析当前页面已经显示出来的商品。
- v1 的“销量”来源于 noon 页面上可见文本，不通过内部接口或隐藏 API 获取。
- v1 建议先用本地规则生成，暂不接入大模型。
- v1 默认支持用户当前主要使用的 noon 页面；如果 noon 不同国家站点 DOM 差异明显，再逐步扩展 selector 配置。
