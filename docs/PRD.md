# Noon Sales Ops Copilot Chrome 插件 PRD v0.1

## Summary

面向 noon 电商平台销售运营人员，做一个运行在商品列表、搜索结果、店铺商品页上的 Chrome 插件。插件通过页面右侧浮窗读取当前页面可见商品信息，按页面可见销售表现指标排序，并基于排序结果给出运营辅助建议，帮助用户快速判断哪些商品值得跟进、哪些商品需要优化、哪些商品可作为选品或竞品参考。

v1 聚焦“当前页面可见商品”的轻量分析，不做账号登录、不连接后端、不跨页自动抓取、不上传数据。

## Product Goals

- 在 noon 当前页面直接完成商品销售表现排序，不要求用户复制粘贴数据。
- 在插件浮窗中展示排序结果，减少离开页面或打开表格的操作。
- 自动识别高销量、低销量、潜力商品，并给出可执行的运营建议。
- 用低成本、快反馈的 v1 为后续价格监控、竞品追踪、批量导出、AI 运营助手打基础。

## Target Users

- Noon 平台卖家运营人员
- 选品和类目运营人员
- 竞品分析人员
- 需要快速浏览商品表现的小团队负责人

## Core User Stories

- 作为运营，我在 noon 商品列表页打开插件后，可以看到当前页面商品按销售表现从高到低排序。
- 作为运营，我可以在浮窗里看到商品名称、价格、销量、评分、排名、商品链接等关键字段。
- 作为运营，我点击浮窗里的商品标题后，页面会定位到当前商品卡片，方便我直接查看页面上下文。
- 作为运营，我可以快速识别当前页面销量最高的商品，并获得“可参考其标题、价格、促销方式”的建议。
- 作为运营，我可以看到低销量但价格或评分有潜力的商品，并获得优化建议。
- 作为运营，我可以一键复制或导出当前分析结果，方便发给团队或放入表格。

## V1 Scope

### Chrome 插件入口

- 使用 Chrome Extension Manifest V3。
- 插件图标点击后，在当前 noon 页面内打开一个右侧浮窗。
- 浮窗可关闭、刷新分析、切换排序方式。
- 仅在 noon 相关域名页面展示可用分析；其他网站显示“不支持当前页面”。

### 当前页面商品识别

插件从当前页面 DOM 中提取当前可见商品卡片信息：

- 商品名称
- 商品链接
- 商品图片
- 当前价格
- 原价或折扣信息，如果页面可见
- 销售量，如果页面可见，例如详情页 `860+ sold recently`
- Ratings 数量，如果页面可见，例如列表页或详情页 `40 Ratings`
- 评分和评论数，如果页面可见
- 当前页面展示位置

默认只分析当前已经加载出来的商品，不自动翻页，不模拟滚动加载。

### 销售表现排序

- 默认按 Ratings 数量代表的热度从高到低排序。
- Ratings 数量是默认排序逻辑，因为 noon 列表页中用户截图标注的商品热度数据通常以 rating/review 数量展示。
- 真实销量或近期销量仍作为独立销售指标保留，例如 `sold recently`，用户可在排序下拉框中切换查看。
- 当使用 Ratings 作为回退指标时，UI 和导出中必须标明来源，不能伪装成真实销量。
- 支持切换：
  - 热度高到低
  - 销量指标高到低
  - 价格低到高
  - 价格高到低
  - 评分高到低
  - 页面原始顺序
- 如果商品缺少销量和 Ratings 字段，标记为“未识别销量”，排在有销售表现指标商品之后。
- 销量文本需要支持常见格式解析，例如 `100 sold`、`100+ sold`、`1.2K sold`、`10k+`、`860+ sold recently`。
- Ratings 文本需要支持常见格式解析，例如 `40 Ratings`、`3.4 (62)`，以及详情页评分旁单独数字格式，例如 `4.7 23`、`4.7 ★★★★★ 23`。
- 如果 noon 详情页延迟渲染或拆分渲染评分和 ratings 数量，插件需要通过评分节点的父级、相邻兄弟节点和整块商品详情文本继续识别热度。
- 商品抽取必须在进入排序列表前完成通用去重，避免同一商品因多个链接、嵌套卡片、追踪参数或详情页重复入口被展示多条。
- 去重维度至少包括 canonical SKU、规范化商品标题、商品标题加价格、图片路径；重复候选需要合并缺失字段，而不是追加新行。
- 商品抽取必须在进入排序列表前完成非商品过滤，避免把 Best Seller、筛选项、促销块、导航块、配送说明等非商品内容作为商品展示。
- 商品候选至少需要具备有效商品标题，并满足商品详情链接、价格、热度、销量、评分、图片等商品信号之一组；纯运营标签或工具性文本需要被拒绝。
- 浮窗排序列表只展示已完成基本字段校验的商品：至少需要可用 noon 商品详情链接和有效商品标题，并具备价格、热度、销量、评分或图片等可辅助判断的字段之一。
- 没有可用商品详情链接的候选不进入展示、复制或导出结果，避免在页面切换或异步加载期间产出不可用链接。

### 自动刷新

- 当浮窗保持打开时，插件必须跟随当前页面变化自动刷新商品清单。
- noon 站内 SPA 跳转、浏览器前进后退、URL hash 变化后，浮窗应自动重新分析当前页面。
- 当前页面商品 DOM 异步更新后，浮窗应自动重新分析，不要求用户每次手动点刷新。
- 用户点击“刷新”后，按钮应立即进入刷新中状态，并展示正在读取、等待页面数据、刷新完成等明确状态，避免用户误以为没有反应。
- noon 站内跳转或点击商品/类目链接后，浮窗应先提示“页面已变化，等待商品数据加载”，并在短时间内多次重试读取，等待页面异步渲染完成。
- 页面切换后，在新页面商品数据尚未稳定前，不继续展示旧页面排序结果；新页面中已具备可用链接和标题的商品可以逐个进入列表。
- 自动刷新应使用渐进式加载体验：展示进度条、当前识别轮次、已显示可用商品数和预估候选商品数，让用户理解加载进度。
- 刷新完成前需要等待结果连续稳定，避免 noon 异步渲染尚未结束时提前把半成品结果标记为完成。
- 站内跳转后的早期读取轮次只用于探测和进度更新，不直接发布商品列表，降低旧页面 DOM 尚未替换时产生错配结果的风险。
- 自动刷新监听需要过滤插件自身 DOM 和明显无关的页面变化，只响应 URL、商品链接、价格、ratings、sold 等商品相关变化，减少卡顿和无效刷新。
- 自动刷新不能重复插入多个浮窗，也不能把插件浮窗自身 DOM 变化当作商品变化无限刷新。
- 如果 ratings 数量延迟出现，自动刷新应在 DOM 更新后重新分析并展示热度。

### 当前页商品定位

- 浮窗商品列表中的商品标题不打开新页面。
- 点击商品标题后，当前页面滚动定位到对应商品卡片或详情页商品区域。
- 被定位的商品区域需要短暂高亮，方便用户确认位置。
- 点击定位时，浮窗需要自动进入临时收起的“定位查看模式”，让出页面右侧视野，避免商品卡片刚好被浮窗遮挡导致用户以为没有反应。
- 定位查看模式结束后，如果浮窗原本是展开状态，需要自动恢复；如果用户原本手动折叠，则保持折叠。
- 如果当前页面已切换导致无法定位，浮窗应给出轻量提示，不应报错。

### 浮窗分析结果

浮窗包含三块：

- 顶部概览：商品数量、识别到销售表现指标的商品数、最高指标、平均价格。
- 排序列表：每个商品展示排名、名称和指标标签；商品名称点击用于当前页定位。
- 商品卡片主高亮标签必须跟随当前排序指标变化：热度排序高亮热度，销量指标排序高亮销量，价格排序高亮价格，评分排序高亮评分，页面原始顺序高亮位置。
- 除主高亮标签外，每个商品卡片展示两个灰色辅助指标；默认展示销量指标和价格，如果主高亮已经是销量或价格，则灰色辅助指标需要避开重复项。
- 运营建议：基于当前页面商品表现生成简短建议。

### 辅助建议规则

v1 使用本地规则，不依赖大模型 API。

- 高销量参考：销量 Top 3 商品，建议观察标题关键词、价格区间、促销标签、主图风格。
- 价格带建议：识别高销量商品集中价格区间，提示当前页主流成交价格带。
- 低销量风险：价格高但销量低的商品，提示检查价格竞争力、主图、标题或促销。
- 评分机会：评分高但销量低的商品，提示可能需要增加曝光或优化关键词。
- 数据不足：如果页面销量和 Ratings 字段都不可见，提示当前页面无法完成销售表现排序，可改用价格和评分作为辅助指标。

### 导出与复制

- 支持复制当前排序结果为表格文本。
- 支持导出 CSV。
- 字段包含：排名、商品名称、销量、销量来源、价格、评分、评论数、商品链接。

## Non Goals For V1

- 不做自动登录或绕过 noon 权限限制。
- 不做跨页批量抓取。
- 不做后台数据库。
- 不做价格历史追踪。
- 不做自动下单、自动改价、自动发布商品。
- 不上传用户浏览数据到服务器。
- 不保证 noon 所有国家站点页面结构一次性全部兼容，先以当前主用页面为准。

## UX Requirements

- 浮窗宽度约 420px，固定在页面右侧。
- 不遮挡页面主要浏览区域过多，可折叠。
- 使用运营工具风格：信息密度高、清晰、少装饰。
- 商品列表要易扫读，排名、销量、价格必须突出。
- 对无法识别的字段明确显示为空或“未识别”，避免假数据。
- 操作按钮包括：刷新、复制、导出 CSV、关闭。
- 刷新操作需要有可感知的进度反馈：等待页面加载、正在读取商品、完成时间、识别商品数。
- 手动刷新和自动刷新完成后，浮窗中需要显示最近更新时间和本次识别商品数量。
- 新页面加载期间使用骨架屏和进度条；已准备好的商品逐个展示，未准备好的商品不以空链接或半成品行展示。
- 商品定位需要避免被右侧浮窗遮挡。优先采用自动临时收起浮窗的方式，而不是要求用户手动拖拽或调整宽度。

## Technical Approach

- Chrome Extension MV3。
- `manifest.json` 定义插件入口和最小权限。
- content script 读取页面商品卡片并注入浮窗。
- background service worker 在点击插件图标时注入脚本和样式。
- shared parser module 解析商品字段和销量文本。
- content script 使用 URL 变化监听和 `MutationObserver` 在浮窗打开期间自动刷新当前页面商品清单。
- content script 对 `pushState`、`replaceState`、`popstate`、`hashchange`、站内链接点击和 URL 轮询做组合监听，提高 noon SPA 跳转后的刷新及时性。
- content script 对商品相关 DOM 变化做轻量过滤和防抖，减少无关页面更新导致的重复解析。
- parser 在排序前执行 `isLikelyProduct` 过滤，并将非商品过滤纳入长期评测集。
- parser 提供 `isDisplayReadyProduct`，content script 只渲染具备可用商品链接和基本业务字段的 display-ready 商品。
- content script 使用结果签名稳定性判断刷新完成；如果商品链接、数量或核心指标仍在变化，继续重试并渐进更新列表。
- 解析结果保留当前页面商品 DOM 引用，仅用于点击浮窗商品标题时滚动定位，不参与导出和持久化。
- content script 在定位商品时临时给面板添加定位状态 class，先让出页面可视区域，再将目标商品滚动到页面中间并高亮。
- 数据处理全部在浏览器本地完成。
- DOM 解析采用可配置 selector 策略，避免把 noon 页面结构硬编码散落在 UI 逻辑中。
- 建议生成采用纯前端规则引擎，后续可替换为 AI 建议接口。

## Acceptance Criteria

- 在 noon 商品列表页点击插件后，可以打开页面内浮窗。
- 插件能识别当前页面可见商品，并展示至少商品名、价格、链接。
- 默认排序按 Ratings 数量代表的热度从高到低排序。
- 如果页面展示 `sold recently`，插件能解析为可切换查看的销售表现指标。
- 如果页面只展示 `Ratings` 数量，插件能解析为销售热度回退指标，并清楚标记来源。
- 如果详情页评分和 ratings 数量拆分到相邻 DOM 节点，插件仍能识别正确热度。
- 无销量和无 Ratings 商品不会导致插件报错，且会被清楚标记。
- 同一个商品不会因为页面内重复链接、嵌套卡片或不同追踪 URL 而在浮窗中重复出现。
- 非商品运营块、导航块、筛选块、促销标签不会进入排序列表。
- 没有可用 noon 商品详情链接的候选不会进入浮窗排序、复制和 CSV 导出。
- 浮窗打开后切换 noon 页面或当前页面商品异步变化时，商品清单会自动刷新。
- 用户点击刷新后，可以看到明确刷新状态、完成时间和识别商品数量。
- 切换不同 noon 页面后，旧页面结果会被清理，新页面已完整识别的商品逐个显示，进度条持续反馈加载进度，直到结果稳定后显示完成。
- 点击浮窗商品标题不会打开新页面，而是在当前页面定位并高亮对应商品。
- 点击浮窗商品标题定位时，浮窗不会遮挡目标商品；用户可以明显看到被定位和高亮的页面商品。
- 商品卡片高亮指标必须真实对应当前选择的排序指标；灰色辅助指标需要避开主高亮指标，不能连续重复展示同一指标。
- 用户可以刷新分析结果。
- 用户可以复制排序结果。
- 用户可以导出 CSV。
- 插件不会修改 noon 页面原始数据或触发购买、登录等敏感操作。
- 在非 noon 页面打开时，插件显示不支持提示。

## Evaluation Policy

每次开发完成前必须运行长期评测集，并产出测试报告。只有 `evals/evaluation_set.json` 中全部用例 100% 通过，且基础自动化测试通过，才算开发完成。

如果开发过程中发现 PRD 与实际实现、noon 页面结构、Chrome 权限模型或用户目标存在冲突，必须先指出冲突并更新 PRD 或实现方案，不能静默偏离 PRD。

## Test Plan

- 使用 noon 搜索结果页测试：确认商品数量、排序、链接、价格识别正常。
- 使用 noon 店铺或类目页测试：确认不同列表结构下浮窗不崩溃。
- 使用无销量展示页面测试：确认出现“数据不足”建议。
- 使用销量文本样例测试解析：
  - `100 sold`
  - `100+ sold`
  - `1.2K sold`
  - `10k+`
  - `860+ sold recently`
  - 空值
- 使用 Ratings 文本样例测试解析：
  - `40 Ratings`
  - `3.4 (62)`
  - `4.7 23 AED 129`
  - `4.7 ★★★★★ 23 | Best Seller`
- 测试 noon 站内跳转和商品 DOM 更新后浮窗会自动刷新。
- 测试页面切换后不会继续展示旧页面商品；只有具备可用商品详情链接的商品才会进入排序列表。
- 测试刷新进度条、骨架屏和已显示/候选商品数量文案存在，且刷新完成前需要结果稳定。
- 测试重复商品候选会真实折叠为单条记录；这个去重用例必须作为长期评测集的上线门禁。
- 测试默认排序使用 Ratings 数量代表的热度，而不是 sold/recent-sold 数量。
- 测试商品卡片主高亮指标跟随排序选择变化，且灰色辅助指标不会重复主高亮指标。
- 测试浮窗商品标题点击会调用当前页定位能力，而不是新开页面。
- 测试定位时会触发临时收起浮窗、居中滚动和高亮目标商品，避免目标被浮窗遮挡。
- 测试复制结果是否能直接粘贴到表格。
- 测试 CSV 是否包含正确列名和 UTF-8 内容。
- 测试非 noon 页面不会注入错误 UI。
- 测试刷新后不会重复插入多个浮窗。

## Assumptions

- v1 只分析当前页面已经显示出来的商品。
- v1 的“销量/销售热度”来源于 noon 页面上可见文本，不通过内部接口或隐藏 API 获取。
